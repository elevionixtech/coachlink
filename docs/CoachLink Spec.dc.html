CoachLink — Class & Client Management Platform

Purpose. CoachLink manages services, clients, instructors, billing, and class operations for businesses that run regular enrolled classes. The initial vertical is fitness and yoga studios; the domain model is deliberately generic so art studios, coaching centers, and clinics can adopt it without schema changes.

This spec describes the system behind the approved prototype (CoachLink.dc.html): data model, business rules, API surface, and a phased build plan. UI behavior follows the prototype unless stated otherwise.

1 · Scope

In scope (v1): service catalogue with deliverables; client CRM with lifecycle, preferences, corporate/family accounts, contact notes; instructor registry; locations, batches, enrollments; subscriptions with percentage discounts; invoice generation from subscriptions; a KPI dashboard.

Also in scope (v1): multi-tenancy, following the shared multi-tenancy specification used across our products (ContractHub): organisations as the tenant boundary, org-code login, a superadmin platform-administration surface, and platform subscription plans with expiry enforcement. There is no self-signup — organisations and members are created only through the platform/admin flows (§5.7).

Out of scope (v1): payments/gateway integration, attendance tracking, fine-grained roles & permissions, client-facing portal, notifications (email/WhatsApp), GST invoice PDFs. These are listed as later phases in §8.

2 · Architecture overview

A conventional three-tier web application is sufficient; nothing in the domain requires exotic infrastructure.

  • Frontend: SPA (React or equivalent) mirroring the prototype's five sections — Dashboard, Clients, Services, Instructors, Operations — plus the global Invoices ledger. Client-side routing per §6.
  • Backend: REST API (§7) over a relational database. All business rules (§5) live server-side; the UI never computes billing on its own.
  • Database: PostgreSQL, shared schema. Top-level domain tables (service, client, instructor, location, batch) carry a NOT NULL org_id (fk → organisation, ON DELETE RESTRICT, indexed). Child tables (deliverables, contact notes, subscriptions, enrollments, invoices) inherit tenancy through their tenant-scoped parent — org_id is not duplicated onto them.
  • Login & tenant resolution: single domain. Login takes three fields, in this order: organisation code · username · password (code mandatory, case-insensitive; the reserved code PLATFORM routes to org-less platform accounts and can never be claimed by a real org). On failure, always the same generic message — never reveal which part was wrong. The session/JWT carries only the user id; the org is resolved server-side from the user row on every request, so nothing tenant-related is ever trusted from the client and org moves/deactivations/lapses take effect immediately.
  • Roles: superadmin — platform operator, the only role allowed org_id NULL (DB CHECK), blocked from all domain APIs by construction; admin — org-level admin (settings and members); staff — org-level regular user. Usernames are unique per organisation.
  • Enforcement: app-layer — a require_org_user dependency on every domain endpoint rejects org-less users (403) and yields the caller's org id; the isolation test suite (§5.6) is the guard. Postgres row-level security may be added as a defence-in-depth backstop.

Per-org configuration lives on the Organisation record (§3.0): currency & locale, invoice grace days, invoice number prefix, capacity policy, and the lookup sets (units, location types, lead sources) — this is what lets a non-fitness vertical rename its vocabulary without code changes. Uniqueness constraints that were global become per-org composites (org_id, business_key) — SKU, location/batch codes, invoice numbers, usernames — so the same SKU may exist in two organisations; case-insensitive name checks are scoped per org too.

Extensibility principle. Nothing fitness-specific is hard-coded: service types, delivery modes, units, location types, and lead sources are lookup enums per organization, so a music school can rename "Studio" to "Practice room" without a migration.

3 · Data model

Three tenancy entities (Organisation, AppUser, SubscriptionPlan) plus eight core domain entities. All tables carry id (uuid), created_at, updated_at; top-level domain tables additionally carry org_id (§2). These are omitted below.

3.0 Organisation, AppUser & SubscriptionPlan (tenancy)

Organisation (the tenant boundary): name (unique) · code (unique, 2–20 chars [A-Z0-9-], stored uppercase — used at login; "PLATFORM" reserved) · plan_id (fk → subscription_plan, nullable) · subscription_starts_on · subscription_ends_on · currency (default INR) · timezone · invoice_prefix (default "INV") · invoice_grace_days (default 7) · capacity_policy (warn | block) · settings (jsonb: units, location types, lead sources, note channels). Each org keeps its own invoice sequence.

AppUser: org_id (nullable fk) · name · username · password_hash · role (superadmin | admin | staff). DB CHECK: org_id IS NOT NULL OR role = 'superadmin'. Composite unique (org_id, username), plus a partial unique index on username WHERE org_id IS NULL so org-less platform accounts stay unique among themselves. Contact notes record their author. Cross-org access is not supported — a person working at two studios has two accounts.

SubscriptionPlan (platform billing — distinct from a client's service subscription, §3.7): name (unique, case-insensitive) · amount (₹) · no_of_days (int > 0) · description. CRUD is superadmin-only; DELETE returns 409 ("In use by N organisations") while any org references the plan, with FK RESTRICT as backstop.

3.6 Enrollment

batch_id (fk) · client_id (fk) · start_date. Unique on (batch_id, client_id). Creatable from either the batch page or the client page — both paths hit the same endpoint.

3.7 Subscription

client_id (fk) · service_id (fk) · start_date · discount_pct (numeric 0–100) · status (active | ended). Effective rate = service.rate × (1 − discount_pct/100), rounded to the rupee.

3.8 Invoice

number (INV-YYYY-NNNN, org-sequential) · client_id (fk) · subscription_id (fk) · period_label · issue_date · amount · status (due | paid | void). Unique on (subscription_id, period_label) — this is what makes generation idempotent. "Overdue" is derived, not stored: status = due AND issue_date + grace_days < today (grace_days is an org setting, default 7).

5 · Business rules

5.1 Lifecycle stage

Stage is staff-editable, with one automatic transition: creating a subscription promotes the client to Customer if not already. Suggested (not enforced) semantics: Lead = expressed interest; Prospect = trial taken or in discussion; Customer = has an active subscription or enrollment; Lapsed = previously Customer, nothing active.

5.2 Invoice generation ("generate missing")

For each active subscription, walk billing periods from start_date to today, stepping by the service's billing interval (Weekly +7d; Monthly +1m; Quarterly +3m; Semi-Annual +6m; Annual +12m). For each period without an invoice for (subscription, period_label), create one with status due at the effective (discounted) rate. Interval N/A produces exactly one invoice, dated start_date.

The operation is idempotent and runs at three scopes: one client (client → Invoices tab), all clients (global Invoices page), and nightly by a scheduled job. Period labels: "Jul 2026", "Q3 2026", "Wk of 14 Jul 2026", "H2 2026", "2026", "One-time".

5.3 Family & corporate accounts

Family links may only point to an existing client; the reverse link is shown automatically on the linked member's profile ("Linked by"). Corporate clients carry company details (name, GSTIN, contact) used on their invoices. A client is exactly one of Individual / Corporate / Family; switching away from Corporate clears company fields, switching away from Family clears the link.

5.4 Contact preferences

The three do-not flags are hard gates for any outbound automation (Phase 3): do_not_contact suppresses all channels; the other two suppress their channel. The UI surfaces flags on the client list and header so staff see them before manual outreach.

5.5 Capacity

A batch's capacity is its location's capacity_per_batch. Behavior follows the org's capacity_policy: warn (default) allows override; block rejects the enrollment. Rosters display fill as "enrolled/capacity".

5.6 Tenant isolation

Every domain endpoint passes through require_org_user. Every list/aggregate filters by the caller's org id; every fetch-by-id verifies ownership — a foreign tenant's id behaves as a 404, never a 403 (no existence leaking). Any foreign key arriving in a request body (service ids, batch ids, family links, …) is verified to belong to the caller's org before being attached, else 404; mutations reached through a child id (a deliverable, a note) walk up to the tenant-scoped parent and verify org there. Backups, exports, and the nightly invoice job iterate per org. Org deletion is a soft close, never a cascade drop.

The isolation test suite is the guard (enforcement is app-layer): tests assert org-scoped lists and stats, foreign-tenant 404s on reads and mutations across every endpoint, rejection of cross-org reference ids, per-org uniqueness (same SKU or username in two orgs is fine; duplicates within one org conflict), the login matrix (wrong code fails, code case-insensitive, reserved code only matches platform accounts), the role matrix (staff can't manage members; org admin can't touch foreign orgs; superadmin manages any org but gets 403 on all domain APIs), and subscription-expiry behavior. Every new endpoint must extend the suite.

5.7 Platform subscriptions & expiry

Assigning a plan (superadmin) stamps starts_on (defaults to today) and ends_on = starts_on + plan.no_of_days. Status is derived on read: none · active · expired. Expiry is enforced by one shared helper at three chokepoints: (1) login — members of an expired org get 403 naming the org and expiry date; (2) require_org_user — checked per request, so tokens issued before the lapse stop working the moment it expires; (3) member management by the org's own admin. The superadmin is never blocked (that is how renewal happens), and renewal restores access instantly with no re-login. Orgs with no plan assigned are not blocked — only a lapsed subscription blocks.

7 · API surface (v1)

REST, JSON, cursor pagination on lists. All domain routes below pass require_org_user — the org comes from the caller's user row on every request, never from a URL or request body; foreign-tenant ids answer 404. Standard CRUD for services, clients, instructors, locations, batches:

GET/POST        /api/{services|clients|instructors|locations|batches}
GET/PATCH/DELETE /api/…/:id

POST   /api/clients/:id/notes            append contact note
GET    /api/clients/:id/invoices
POST   /api/clients/:id/subscriptions    { service_id, start_date, discount_pct }
POST   /api/enrollments                  { client_id, batch_id, start_date } → 409 on duplicate
GET    /api/invoices?status=&client_id=
POST   /api/invoices/generate-missing    { client_id? }  → { created: n }   (idempotent)
PATCH  /api/invoices/:id                 { status: "paid" | "void" }
GET    /api/dashboard                    KPI aggregates

POST   /api/login                        { org_code, username, password } → session (user id only)
GET/PATCH /api/org                       current org settings (org admin)
GET/POST  /api/organisations/:id/members own admin or superadmin; staff → 403, foreign org → 404;
                                         created members: admin | staff, never superadmin
POST   /api/admin/organisations          superadmin: org (name + code) + first user, role forced admin
GET    /api/admin/organisations          superadmin: list with member counts + subscription status
CRUD   /api/admin/plans                  superadmin; DELETE → 409 while any org references the plan
POST   /api/admin/organisations/:id/plan assign plan → stamps starts_on / ends_on

Validation errors return 422 with per-field messages mirroring the modal's "Required: …" pattern. All list endpoints accept q (search) where the UI offers it.

9 · Open questions

  • Should family-linked clients share a single invoice (one payer per family) or remain individually billed with the Family Plan discount?
  • Proration: when a subscription starts mid-period, is the first invoice full-rate or prorated?
  • Does a Usage Based service (class pack) need per-visit deduction in v1, or is that part of Phase 3 attendance?
  • Is capacity a hard cap (block enrollment) or soft warning by default?
  • GST treatment: are services taxed uniformly, and does the org need invoice-level GSTIN capture beyond corporate clients?
